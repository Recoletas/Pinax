# 当前执行计划

> 本文件是继续推进时的默认入口。先看这里，再决定是否打开专题页。当前阶段不再按“诗歌页还差什么、散文页还差什么”来排期，而是按工具链缺口来排期。

## 1. 当前结论

粗略进度：88%

已完成的不是零散功能，而是四块基础设施：

1. 会话和运行态边界已经稳定。
2. 世界书上下文和素材中台已经形成闭环。
3. 生成调用已经统一到任务层。
4. 诗歌、散文、体验、顾问都已经纳入统一生成体系。

因此，真正剩下的只剩四条线：

1. 编辑器与统一智能顾问收口。
2. 记忆系统收尾。
3. 统一分镜输出层。
4. 产品叙事和信息架构从“功能页”切到“工具链”。

## 3. 当前执行顺序

### 3.1 先完成：规划语言和边界冻结

这一步已经通过本轮文档重写启动，目标是先把“项目到底是什么”说清楚，避免后续继续按旧页面心智新增实现。

当前要求：

- 文档统一使用“工具链、素材、设定、记忆、分镜”的主语言。
- 诗歌、散文在规划层不再被写成两个产品中心。
- 所有剩余任务必须能挂到工具链阶段，而不是只挂到某个页面。

### 3.2 当前主线 A：编辑器与统一智能顾问

优先级最高，因为这是当前最明显的用户痛点，而且它会直接影响写作主链路。

已确认的现状：

1. 写作页不是 CodeMirror 或 ProseMirror，而是 textarea + ghost overlay。
2. 自动补全在用户输入停顿后高频触发，当前目标是“直接续写正文”，这对稳定性要求过高。
3. 顾问组件仍保留 AI / OpenClaw 双模式，但页面调用并未真正把 OpenClaw 回调接通，说明这条线本身也需要收口。

当前结论：

- 不直接上 ProseMirror。
- 不把 CodeMirror 6 作为第一步。
- 第一阶段优先保留当前文本编辑器，改为 OpenClaw 统一智能顾问，主打自动收线、纠错、局部修复和章节体检。
- 自动补全保留为弱能力，只在显式触发或低频场景下存在。
- OpenClaw Gateway 不可用时要有明确错误和降级说明；第一阶段不再把后端切换暴露给用户。

当前应看专题：[editor-advisor-unification.md](./editor-advisor-unification.md)

### 3.3 当前主线 B：记忆系统收尾

这是编辑器与统一顾问之后的下一优先级，因为它已经接近完成，而且会影响体验和写作的长期稳定性。

本线只做四类事：

1. 候选批量整理。
2. 本地与远端记忆去重。
3. 冲突表达和确认策略。
4. 同步状态和失败重试的可见性。

不做的事：

- 不把记忆继续做成自动写入系统。
- 不把记忆和世界书混成同一种资产。
- 不在页面里零散加同步提示，先确定统一状态表达方式。

### 3.4 当前主线 C：统一分镜输出层

这是下一块真正新的系统性工作，目标是把诗歌、散文、体验片段和章节片段统一收束到同一套 storyboard 结构。

本线分三步：

1. 先定义 schema、版本对象和校验规则。
2. 再做适配层，优先接散文 directing cards，再接诗歌 directing tree，最后接体验素材和章节片段。
3. 最后再做导出和界面迁移，不反过来。

不做的事：

- 不先做 UI 大改后补底层对象。
- 不继续扩诗歌和散文各自的独立分镜格式。
- 不把导出做成页面私有功能。

### 3.5 后续主线 D：产品信息架构收口

等记忆系统和分镜核心对象有了稳定落点后，再做导航和页面入口收口。

方向：

- 一级导航优先体现体验、写作、素材、设定、分镜、顾问。
- 诗歌和散文作为写作/分镜下的模式、模板或入口存在。
- 用户最终看到的是连续工作流，而不是多个互不相干的小工具页。
