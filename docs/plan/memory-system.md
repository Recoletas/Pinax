# 05 - 记忆候选与长期知识层

状态：收尾中

## 1. 在工具链中的位置

记忆系统负责处理“值得跨会话复用，但又不应该直接写进世界书”的长期上下文。

标准路径应当是：

```text
体验 / 写作内容
	-> 记忆候选抽取
	-> 作者确认和 scope 选择
	-> 本地记忆 / 远端 mem0 同步
	-> 后续体验或写作读取
```

这条线的目标不是“自动更聪明”，而是“长期上下文可控、可解释、可同步”。

## 2. 和记忆容易混淆的对象边界

| 内容类型 | 正确去向 | 原因 |
|----------|----------|------|
| 稳定的世界设定、角色设定、地点设定 | 世界书草稿 / 世界书 | 它属于作品设定层 |
| 作者长期偏好、项目约束、反复出现的创作要求 | 记忆系统 | 它属于长期上下文，而不是设定条目 |
| 当前会话临时状态 | Session / RuntimeState | 它不应污染长期层 |
| 暂时不确定的信息 | 素材或候选 | 先保留，后确认 |

## 3. 已完成的基础能力

- 记忆写入从静默写入改为候选确认流。
- 候选支持 `global-author`、`project`、`session` 三类 scope。
- 候选面板支持作用域筛选、计数、编辑后确认。
- 已确认本地记忆会注入小说体验生成上下文。
- mem0 写入适配已接入，未配置时自动跳过。
- mem0 读取兜底已接入，本地无确认记忆时按 scope 搜索远端记忆。
- `useMem0` 已支持 scope metadata、scope 检索和缓存隔离。

## 4. 当前剩余缺口

| 缺口 | 当前问题 | 目标状态 |
|------|----------|----------|
| 候选批量整理 | 候选多时操作成本高 | 已支持批量确认/拒绝/改 scope/归档，已归档视图可恢复，排序筛选待后续 |
| 本地 / 远端去重 | 明显重复和保守相似已可提示，远端精确重复可跳过写入 | 后续补更强语义去重 |
| 冲突策略 | 新候选与已有记忆冲突时表达不够清楚 | 已有基础冲突提示、替换旧记忆和合并旧记忆，后续再补对照 |
| 同步状态表达 | 已做最小可见性，但仍可继续细化批量与聚合 | 用户能看见同步状态与重试入口 |

建议先补清楚数据字段口径，再做 UI。以下字段可以先作为规划约束记录，不要求一次性全部实现：

| 字段 | 用途 |
|------|------|
| `syncStatus` | 区分 `local-only`、`pending`、`syncing`、`synced`、`failed` |
| `remoteId` | 记录 mem0 或其他远端系统返回的 ID |
| `lastSyncedAt` | 显示最近一次成功同步时间 |
| `lastSyncError` | 保留最近失败原因，供 UI 展示和重试 |
| `contentHash` | 做保守去重和本地变更检测 |
| `duplicateOf` | 标记疑似重复目标，当前已用于完全重复提示 |
| `similarTo` | 标记保守规则下的相似重复目标 |
| `conflictsWith` | 标记冲突目标列表 |

## 5. 执行方案

### 5.1 M1 - 候选整理面板收口

目标：先把“候选太多不好处理”的问题解决。

建议交付：

- 批量选择。
- 批量确认。
- 批量丢弃或归档。
- 批量改 scope（已接入，后续补默认值和批量筛选细化）。
- 批量归档和已归档视图（已接入，排序重排后续补）。
- 按来源、scope、时间排序或筛选。

验收：

- 作者可以在一次操作里处理一组候选，而不是逐条点选。
- 面板能清楚显示候选来源和当前建议 scope。

当前进展：

- 待确认候选已支持批量选择、全选可见、批量确认、批量拒绝、批量改 scope 和批量归档。
- 已归档视图已接入，归档后可恢复为待确认或已确认；排序重排和更强的整理规则后续补。

### 5.2 M2 - 去重与冲突处理

目标：减少长期层污染，而不是简单多存几条。

建议策略：

1. 先做规则型去重，不直接上复杂语义聚类。
2. 以标准化文本、scope、项目归属、来源相似度做基础判重。
3. 对“疑似重复”与“明显冲突”分开处理。

建议输出：

- 疑似重复提示。
- 已有记忆对照视图。
- 合并、保留两条、替换旧条三种操作。

验收：

- 新候选进入前，能发现同 scope 下的高相似项。
- 冲突时作者能明确决定保留、合并还是替换为新记忆。

当前进展：

- 已实现保守规则型去重：同 `scope + scopeId + kind` 且标准化内容完全一致时标记 `duplicateOf`。
- 已实现本地相似提示：同 `scope + scopeId + kind` 且内容高度相似时标记 `similarTo`。
- mem0 同步前会做远端精确重复检查，已存在完全相同远端记忆时复用远端 ID，不重复写入。
- 已实现基础冲突提示：同 `scope + scopeId + kind` 且内容不同、存在已确认记忆时会标记 `conflictsWith`。
- 冲突候选已支持一键替换旧记忆或合并旧记忆，旧记忆会本地标记为 `stale`。
- 更强语义相似、远端合并/删除和已有记忆对照视图后续补。

### 5.3 M3 - 同步状态与失败恢复

目标：让 mem0 不再像黑盒后台动作。

建议交付：

- 每条记忆或每批操作都有同步状态。
- 明确区分未同步、同步中、同步成功、同步失败。
- 失败时能重试，且不影响本地已确认状态。
- 远端不可用时，界面给出降级解释，不阻塞本地流程。

验收：

- 用户能知道哪些记忆只在本地，哪些已完成远端同步。
- 远端失败不会把本地状态打乱。

当前进展：

- 记忆候选面板已增加“待确认 / 已确认”视图。
- 已确认项会显示 `local-only`、`syncing`、`synced`、`failed` 状态。
- 失败或仅本地状态可直接重试同步。

### 5.4 M4 - 注入可解释性补强

这一步不是阻塞项，但在分镜主线启动前最好完成最小可见性。

建议至少做到：

- 生成时能知道本次读到了哪些确认记忆。
- 不同 scope 的读取结果可区分。

## 7. 风险控制

- 不要把“相似”误判成“应合并”，去重宁可保守。
- 不要把世界书事实和记忆事实混放到同一确认流里。
- 不要把远端同步失败做成本地确认失败。

## 8. 测试要求

2. 候选批量操作测试。
3. 去重 / 冲突策略测试。
4. 同步失败和重试测试。
