# 叙事工具链总计划

> 后续推进默认只看本文件和 [plan/current-execution-plan.md](./plan/current-execution-plan.md)。专题文件只在需要实现细节、边界定义或验收口径时打开。

## 1. 当前判断

项目已经不适合继续按“诗歌页 + 散文页 + 体验页”这种功能并列方式理解。

现在真实已经落地的，是一套逐步成形的叙事创作工具链：

```text
设定 / 世界书 / 角色 / 约束
	-> 体验生成与探索
	-> 素材沉淀与整理
	-> 写作加工与续写
	-> 候选设定 / 记忆确认
	-> 分镜化与导出
```

因此后续规划必须遵守两条总原则：

1. 诗歌和散文不再作为产品中心继续扩张，而是作为工具链中的专用生成入口或加工模式。
2. 剩余工作不再是“补几个页面功能”，而是把长期资产层和结构化输出层收口完成。

## 2. 当前总状态

| 工作流模块 | 状态 | 当前结论 |
|------------|------|----------|
| 会话与运行态边界 | 完成 | 主体验链路已稳定，后续只做回归修 bug |
| 世界书上下文 | 完成 | 上下文构建、命中预览、草稿回写边界已建立 |
| 素材中台 | 完成 | 体验到写作的沉淀与消费闭环已形成 |
| 统一生成任务层 | 完成 | 业务生成调用已收口到 task 化服务，页面不再各自走底层生成 |
| 编辑器与智能顾问 | 待启动 | 当前写作页是 textarea + ghost overlay，自动补全问题在任务语义和交互形态上都过重；计划统一到 OpenClaw 智能顾问，主打自动收线与纠错，补全降级为弱能力 |
| 记忆系统 | 基本完成 | 候选、scope、确认、mem0 同步/读取已接入，剩余是整理、冲突、状态表达 |
| 统一分镜输出层 | 待启动 | 这是剩余最大块，需要把诗歌/散文/体验片段统一成 storyboard 资产 |
| 产品叙事与信息架构 | 待收口 | 页面入口仍有旧叙事，需要逐步改成工具链导向 |

## 3. 工具链中的核心对象

后续所有新功能都必须能明确挂到这些对象之一，不能再以页面私有结构充当事实来源。

| 对象 | 含义 | 当前状态 |
|------|------|----------|
| Session | 一次体验会话的容器，负责恢复、切换、保存 | 已稳定 |
| RuntimeState | 会话内运行态，如消息、地图、活动、角色、阶段状态 | 已稳定 |
| NarrativeAsset | 内容流转的中间资产，承接体验、笔记、章节和候选设定 | 已使用 |
| WritingProject / Chapter | 正式写作载体，消费素材并产出正文 | 已使用 |
| WorldbookEntry / WorldbookDraft | 正式设定与待确认设定 | 已使用 |
| MemoryCandidate / MemoryRecord | 待确认记忆与已确认长期记忆 | 已使用，待收尾 |
| GenerationTask | 所有 AI 调用的统一任务抽象 | 已使用 |
| StoryboardDocument / StoryboardVersion | 统一分镜输出对象 | 待建立 |
| AdvisorTask / AdvisorResult | 统一智能顾问任务和结构化结果 | 待建立 |

## 4. 模块重新定义

后续文档、导航和实现都应按下面的含义对齐：

| 模块 | 在工具链中的角色 |
|------|--------------------|
| 体验 | 负责探索、互动、生成剧情和收集原始材料 |
| 素材 | 负责沉淀、筛选、转流向，是工具链的交换层 |
| 写作 | 负责章节加工、续写、改写和结构组织 |
| 设定 | 负责世界书、角色设定、约束和回写边界 |
| 记忆 | 负责长期上下文候选与确认，不直接替代世界书 |
| 分镜 | 负责结构化输出、版本化和导出 |
| 顾问 | 负责跨阶段审阅、建议和辅助分析 |
| 诗歌 / 散文 | 负责提供特定风格文本与分镜生成入口，不再作为独立产品中心 |

## 5. 当前剩余主任务

后续只剩四条真正值得推进的主线：

1. 编辑器与统一智能顾问收口。
当前最大用户痛点是写作页内联自动补全。现状不是单纯“渲染不够高级”，而是基于 textarea 的 ghost 补全和“直接续写正文”这套任务定义本身不稳定。应改为统一 OpenClaw 智能顾问，优先做自动收线、纠错和局部修复，弱化自动补全。

2. 记忆系统收尾。
记忆已经能候选、确认、分 scope、同步和读取，但还缺批量整理、冲突处理和更清楚的同步状态。

3. 统一分镜输出层。
这是剩余最大结构化输出块，需要把诗歌、散文、体验片段和章节片段统一映射到同一套 storyboard schema、版本和导出校验。

4. 产品叙事与信息架构收口。
即使底层已经像工具链，导航和页面命名仍带着旧的功能页叙事，需要逐步把“诗歌/散文”降级为模式或入口，把“素材/设定/分镜”抬为主线能力。

## 6. 阅读规则

1. 总判断先看本文件。
2. 当前任务、顺序、优先级看 [plan/current-execution-plan.md](./plan/current-execution-plan.md)。
3. 写作编辑器和统一顾问方案先看 [plan/editor-advisor-unification.md](./plan/editor-advisor-unification.md)。
4. 记忆相关细节看 [plan/memory-system.md](./plan/memory-system.md)。
5. 分镜统一方案看 [plan/storyboard-system.md](./plan/storyboard-system.md)。
6. 已完成基础能力的边界和验收定义，按需查会话、世界书、素材、生成服务专题页。
7. 最近一次真实落地情况看 [LOG.md](./LOG.md)。

## 7. 非协商规则

- 不新增第二套体验状态源，短期仍以主体验 store 为准。
- 不新增页面层直接调用 `sendChat()` / `sendChatStream()`。
- 不新增页面层直接调用 `runGenerationRetryPlan()`，优先抽专用 service，再走 `runGenerationTask()` / `runGenerationStreamTask()`。
- 不新增页面层直接读写 `localStorage`，统一走 service 或 store。
- 不自动写入世界书和长期记忆，必须先经过候选或显式确认。
- 不在诗歌、散文、体验页内部各自长出独立分镜 schema。
- 所有用户可感知改动都必须能说明“用户在界面上做什么会看到变化”。
