# 开发规范与协作约定

## 1. 代码分层约束

- `pages`：负责流程编排与交互状态，不承载底层通信细节。
- `stores`：负责跨页面状态、业务动作与持久化调度。
- `services/composables`：负责 API 封装与可复用逻辑。
- 组件命名、路由命名、Store action 命名应表达业务语义，避免“临时名”。

## 2. AI 调用约束

- 业务侧禁止直接调用或导入 `sendChat` / `sendChatStream`。
- 页面层不直接调用 `runGenerationRetryPlan`；优先抽专用 `services/*`，再通过 `runGenerationTask` / `runGenerationStreamTask` 执行生成。
- 每个生成任务必须明确：
  - `taskType`。
  - 尝试顺序（first/retry）。
  - 结构化解析策略。
  - 失败提示与兜底策略。
- 结构化输出必须经过字段归一化后再写入 Store。

## 3. UI 可用性基线

当前阶段的 UI 判断以用户可用性为第一优先级。兼容检查用于发现遮挡、滚动锁死、按钮不可点等阻塞问题，不用于要求保留旧入口、旧位置或旧交互层级。

### 3.0 决策顺序

```text
任务可完成 > 入口清晰 > 页面一致 > 移动端不阻塞 > 旧入口兼容 > 视觉细节
```

### 3.1 布局与视口

- 页面级容器优先使用可适配视口单位（如 `100dvh`），避免单纯依赖 `100vh`。
- 不在同一页面叠加多套滚动容器，主滚动区只保留一处。
- 移动端键盘弹出场景必须保证主输入区可见、可编辑。

### 3.2 浮层与定位

- 单页面最多保留一个主悬浮入口；新增浮层前必须先评估是否可并入现有入口。
- 弹层必须定义统一的打开/关闭与焦点回收行为。
- 禁止通过 `!important` 叠加修补层级冲突；优先收口结构再调层级。

### 3.3 交互一致性

- 同类操作在不同页面必须保持相同命名和位置语义。
- 高频任务（导入、保存、生成、应用）目标为 3 步内可达。
- 默认状态优先展示主任务，低频操作收纳到次级区。
- 重复入口优先合并；如果旧入口降低主路径清晰度，可以降级或移除。

### 3.4 响应式回归基线

- 必测宽度：360、768、1280。
- 必测主题：亮色与暗色。
- 必测链路：输入、滚动、弹层、关闭、提交反馈。

## 4. 导入与数据安全

- 任何导入流程都应先预览，再执行写入。
- 同名冲突必须要求用户显式决策，不做静默覆盖。
- 覆盖操作默认视为高风险动作，需有可见提示。

## 5. 提交前检查

执行：
- `npm run verify`

最低通过标准：
- Vitest 全绿。
- 构建通过。
- 无新增 lint/架构守卫错误。

## 6. 文档同步规则

发生以下变更时必须更新 docs：
- 用户可见流程变化（入口、按钮、文案、策略）。
- 数据模型变化（字段、结构、兼容映射）。
- 架构约束变化（调用路径、守卫规则）。
- UI 行为变化（响应式断点、浮层策略、交互路径）。

建议同步顺序：
1. 更新对应分区文档。
2. 追加 `LOG.md` 里程碑记录。
3. 必要时调整 `PLAN.md` 阶段目标描述。
