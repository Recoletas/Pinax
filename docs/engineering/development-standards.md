# 开发规范与协作约定

## 1. 代码分层约束

- `pages`：负责流程编排与交互状态，不承载底层通信细节。
- `stores`：负责跨页面状态、业务动作与持久化调度。
- `services/composables`：负责 API 封装与可复用逻辑。
- 组件命名、路由命名、Store action 命名应表达业务语义，避免“临时名”。

## 2. AI 调用约束

- 业务侧禁止直接调用或导入 `sendChat`。
- 统一使用 `runGenerationRetryPlan`，并明确：
  - 尝试顺序（first/retry）。
  - 结构化解析策略。
  - 失败提示与兜底策略。
- 结构化输出必须经过字段归一化后再写入 Store。

## 3. 导入与数据安全

- 任何导入流程都应先预览，再执行写入。
- 同名冲突必须要求用户显式决策，不做静默覆盖。
- 覆盖操作默认视为高风险动作，需有可见提示。

## 4. 提交前检查

执行：
- `npm run verify`

最低通过标准：
- Vitest 全绿。
- 构建通过。
- 无新增 lint/架构守卫错误。

## 5. 文档同步规则

发生以下变更时必须更新 docs：
- 用户可见流程变化（入口、按钮、文案、策略）。
- 数据模型变化（字段、结构、兼容映射）。
- 架构约束变化（调用路径、守卫规则）。

建议同步顺序：
1. 更新对应分区文档。
2. 追加 `LOG.md` 里程碑记录。
3. 必要时调整 `PLAN.md` 阶段目标描述。
