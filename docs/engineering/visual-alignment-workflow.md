# 视觉对齐工作流

本文定义 Pinax 前端视觉任务的协作流程。视觉任务包括页面构图、控件位置、材质、动效、角色立绘排布、背景融合、响应式适配和整体气质。

核心原则：早期少文字、多截图、小切片、强约束；方向未证明前，不做整页大改。

## 用户标注语义

用户提供 direct、红笔、橡皮擦或截图标注时，按以下规则解释：

- 红框 = 必须落入的位置范围。
- 红线 = 必须跟随的角度、透视或曲线方向。
- 红圈 = 视觉焦点或需要重点处理的对象。
- 橡皮擦 / 擦除区域 = 不应被主要 UI 占用。
- 未标注区域 = 不主动重设计，除非为完成标注目标必须调整。
- 用户写下的“不对”“太平”“像 PPT”“不适配” = 需要转化成可验证的视觉约束，而不是直接让 worker 自由发挥。

## 硬约束

- MUST: 视觉任务开始前，Codex 必须把用户反馈翻译成 5-8 条可验证硬约束。
- MUST: 有 direct / screenshot 标注时，标注优先级高于文字推测和历史实现。
- MUST: 视觉早期必须先做小切片 prototype，不得直接全页重构。
- MUST: 每轮视觉实现后必须给出截图路径或说明无法截图的原因。
- MUST: Codex 必须按截图、标注和用户反馈验收，而不是只看代码 diff。
- MUST: 视觉任务应拆为背景、标题、CTA、右上控件、角色 strip、信息文字、动效等独立切片。
- MUST NOT: 不允许多个 worker 同时改同一个视觉区域或同一核心样式文件。
- MUST NOT: 不允许 worker 未经明确要求主动重做未标注区域。
- MUST NOT: 不允许在方向未确认前写长 spec、长调研或整页实现计划。
- MUST NOT: 不允许只用“倾斜一点”“高级一点”“更细节”等模糊词作为实现 brief；必须落到位置、角度、层级、比例、形状、材质或动效。

## 标准流程

1. 用户提供参考、截图或 direct 标注。
2. Codex 提炼硬约束。
3. Codex 拆 3-5 个视觉切片。
4. 每个 Claude worker 只做一个切片，或 Codex 亲自做高风险精修。
5. worker 交付截图、summary、diff、测试结果。
6. Codex 合并可用部分，丢弃不合格部分。
7. 用户按评分反馈。
8. Codex 把低分项转成下一轮硬约束。

## 视觉切片示例

- 标题嵌入背景弧度。
- 左下 CTA 透视位置与不规则形状。
- 右下角色立绘按小端圆弧排布。
- 右上控制组的斜向层级、描边、材质和 hover 状态。
- 背景动态壁纸：只让局部元素动，不让整图漂移。
- 介绍文字跟随同一透视面，不悬浮成 PPT。

## 反馈评分格式

用户验收时优先用短评分：

```text
标题嵌入：3/5，角度对了但弧度没贴背景
CTA 位置：2/5，还没进 red box
CTA 透视：1/5，缺近大远小
角色 strip：4/5，方向对，需更靠边
右上控件：3/5，倾斜对，细节不足
```

Codex 必须把评分低的项转成下一轮 brief，例如：

```text
CTA 必须移动到 direct 新红框内。
近端按钮 scale 1.08，远端按钮 scale 0.88。
整体旋转方向为向下，不得使用向上角度。
按钮外形使用 clip-path 不规则四边形，不使用普通圆角矩形。
```

## 何时使用 Claude Worker

适合派 Claude：

- 单个视觉区域边界明确。
- 有 screenshot / direct 可对照。
- 文件范围小。
- 可以用截图和契约测试验收。

不适合派 Claude：

- 用户刚刚给出高密度 taste feedback。
- 需要判断多处视觉冲突。
- 需要合并多个 worker 的 CSS。
- 需要最终微调到“差一点点”的阶段。

这些情况应由 Codex 亲自处理，或者只派一个 worker 做候选后由 Codex 手工整合。

## 用户需要提供什么

用户不需要写长需求。优先提供：

- 一张当前页面截图。
- 一张 direct / 红线标注。
- 1-3 个最不满意点。
- 参考项目或参考截图。
- 每轮 1-5 分评分。

用户反馈越短越好，但必须指出位置或对象。比如“右下角色 strip 再靠边，按我画的小端圆弧排”比“整体再高级一点”更有效。
